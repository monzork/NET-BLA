# Generative AI Tool Usage & Reflection

This document reviews how Generative AI tools were used to build and scaffold the Task Management application, highlighting validation processes, corrections, rejected suggestions, test-driven development, and architectural compliance with Clean Architecture and Domain-Driven Design (DDD) principles.

---

## 📖 Informal User Story & Acceptance Criteria

### User Story
> **As a** software developer / project manager,  
> **I want to** securely register and log into a dashboard, where I can manage my project tasks (create, read, update, delete) with due dates and statuses,  
> **So that** I can track my project progress and milestones in real-time.

---

## 💬 Prompt Engineering & Scaffold Generation

### The Generation Prompt
The prompt used to scaffold the backend and core business logic was structured as follows:

```text
Build a RESTful API for a simple Task Management System in .NET 10 C# adhering to Clean Architecture principles.
We have four layers: Domain, Application, Infrastructure, API.
Constraints:
1. DO NOT use Entity Framework Core, Dapper, or MediatR.
2. Use raw ADO.NET with parameterized SQL queries via Microsoft.Data.SqlClient.
3. Use Test-Driven Development. I want to build business services first, and write xUnit unit tests verifying validation rules.
4. Validations: Task Title is required, DueDate cannot be in the past, Status must be Pending, InProgress, or Completed.
5. Create user register/login endpoints. Secure user passwords with hashing and use JWT authorization.
6. Seed an admin user and 12 project-implementation tasks on database startup.
```

## 🔍 Validation of AI Suggestions

AI-generated recommendations were validated against three criteria: **security parameterization**, **architectural boundaries**, and **security code review standards**.

### 1. SQL Injection Prevention (Query Parameterization)
To ensure safety against SQL injection, every database interaction was verified to use parameterized queries rather than string interpolation.
* **Verified Code Example** ([`UserRepository.cs`](file:///home/monzork/code/NET-BLA/src/Infrastructure/Data/UserRepository.cs)):
  ```csharp
  using var command = connection.CreateCommand();
  command.CommandText = "SELECT * FROM Users WHERE Email = @Email";
  command.Parameters.AddWithValue("@Email", email);
  ```

### 2. Strict Architectural Boundaries
Verified that the core domain entities ([`User.cs`](file:///home/monzork/code/NET-BLA/src/Domain/Entities/User.cs), [`TaskItem.cs`](file:///home/monzork/code/NET-BLA/src/Domain/Entities/TaskItem.cs)) and application logic ([`TaskService.cs`](file:///home/monzork/code/NET-BLA/src/Application/Services/TaskService.cs)) do not reference HTTP-specific namespaces (`Microsoft.AspNetCore.Http` or `Mvc`) or database-specific namespaces (`Microsoft.Data.SqlClient`). All persistence is accessed via interface contracts.

### 3. Security Code Review
AI-generated authentication and security structures were verified to comply with standard security checklists:
* **JWT Signing Configuration**: Verified signing keys are pulled from environment configurations and validate issuer, audience, and signing credentials.
* **Token Expiration Handling**: Confirmed tokens use temporary durations (1 hour) and check time limits.
* **Authorization Scope**: Inspected controller routes to verify that endpoints modifying data are marked with the `[Authorize]` attribute.
* **Profile Boundary Access**: Verified that services validate ownership (`UserId` matching current user claims) on every write or retrieve action to prevent cross-tenant exposure.

## 🚫 AI Suggestions Rejected

Several generated suggestions were intentionally rejected or rewritten:

### Example: Repository Leakage into Application Layer
An early version generated by the AI introduced SQL-specific query statements directly inside the application service layer. This violated Clean Architecture principles by coupling the Application layer with database persistence. The code was immediately refactored so that all database operations occur exclusively through abstraction interfaces defined in the Application layer and implemented in the Infrastructure layer.

## ⚡ Corrections & Improvements Made to AI Code

The following key modifications and design improvements were introduced over the raw AI output:

### 1. Decoupling the Clock (`IDateTimeProvider`)
* **The Issue**: The original AI implementation directly referenced `DateTime.UtcNow` inside the business services. This introduced tight temporal coupling, making validations dependent on the system clock and introducing potential race conditions during fast unit test execution.
* **The Solution**: Created the [`IDateTimeProvider`](file:///home/monzork/code/NET-BLA/src/Application/Common/Interfaces/IDateTimeProvider.cs) interface and registered the runtime implementation [`DateTimeProvider`](file:///home/monzork/code/NET-BLA/src/Infrastructure/Services/DateTimeProvider.cs) in the DI container. This allowed tests to inject a mock time provider with a frozen clock value, enabling completely deterministic due date validation and eliminating arbitrary time-tolerance hacks.

### 2. Password Security & Cryptography Swapping
* **The Issue**: The AI initially scaffolded simple SHA256 hashing. Because SHA256 is designed to be fast, it is highly vulnerable to brute-force and dictionary attacks (via GPUs) in production.
* **The Solution**: Decoupled password hashing logic by introducing the [`IPasswordHasher`](file:///home/monzork/code/NET-BLA/src/Application/Common/Interfaces/IPasswordHasher.cs) interface contract in the Application layer, and implemented it as [`BCryptPasswordHasher`](file:///home/monzork/code/NET-BLA/src/Infrastructure/Security/BCryptPasswordHasher.cs) in the Infrastructure layer. This utilizes `BCrypt.Net-Next` with a work factor of `11`, providing computational slowness and resistance against offline brute-force attacks.

### 3. API Protection & Rate Limiting
* **The Issue**: The AI generated open REST endpoints without any rate-limiting logic, exposing the system to denial of service (DoS) and authentication brute-forcing.
* **The Solution**: Configured ASP.NET Core's native `Microsoft.AspNetCore.RateLimiting` middleware. 
  - **Auth Endpoint Protection**: Configured a `fixed window` limiter policy (`auth`) restricted to 10 attempts per minute on login/registration routes.
  - **CRUD Endpoint Protection**: Configured a `sliding window` limiter policy (`global`) allowing 100 requests per minute with segment splitting.

### 4. Startup Database Schema Setup & DbUp Migrations
* **The Issue**: The AI seeder connected to `Database=TaskDb` directly with embedded raw SQL schema strings. On container runs, SQL Server threw database connection failure errors, and incremental migrations were not trackable.
* **The Solution**:
  - Rewrote the [`DbSeeder.cs`](file:///home/monzork/code/NET-BLA/src/Infrastructure/Data/DbSeeder.cs) logic to connect to `master` first, run database creation if missing.
  - Integrated `DbUp` to execute structured SQL scripts as embedded resources for incremental schema tracking (`0001_InitialSchema.sql`, `0002_AddRevokedTokensTable.sql`), separating migration logic from seed data.

### 5. JWT Logout Session Revocation (Blocklist)
* **The Issue**: Once issued, JWT tokens are stateless and remain valid until expiration, presenting a security vulnerability if the user logs out.
* **The Solution**:
  - Introduced a parameterized database-backed token blocklist repository (`IRevokedTokenRepository`).
  - Intercepted token validation events inside `Program.cs` via ASP.NET Core `JwtBearerEvents.OnTokenValidated` to verify the request token is not blocklisted.

## 🏛️ Clean Architecture & DDD Compliance Analysis

### 1. Clean Architecture Compliance
The project is a textbook implementation of Clean Architecture, adhering strictly to the **Dependency Rule** (dependencies point inwards towards the Domain):
* **Domain**: Holds pure objects (`User.cs`, `TaskItem.cs`) with **zero dependencies** on external packages, libraries, or SQL namespaces.
* **Application**: Holds business workflow logic and repository contracts.
* **Infrastructure**: Implements database repository queries, seeders, and cryptography algorithms.
* **API**: Maps controllers and routing endpoints.

### 2. Domain-Driven Design (DDD) Analysis
* **Ubiquitous Language**: The domain models use explicit terms matching the use cases (`User`, `TaskItem`, `TaskItemStatus`).
* **Entities**: Both `User` and `TaskItem` represent Entities as they carry identity attributes (`Guid Id`) that track them uniquely over time.
* **Rich Domain Model (DDD Implementation)**:
  - Refactored the domain layer from an Anemic Model to a Rich Domain Model. Properties are encapsulated with `private set` to protect state invariants.
  - Moved creation validations (such as blank task titles or past due dates) into constructors.
  - Expose intent-revealing domain methods (like `Update` and `Complete`) to manage status transition logic within the domain boundary.
  - Implemented `TaskItem.CreateFromDatabase` static factory to allow database restoring without triggering creation invariants.

## 🧪 Test-Based Validation

AI-generated code was not accepted solely because it compiled. Business requirements were verified through xUnit test coverage covering:
* **Task creation validation**: Blank titles, future due dates, and valid status states.
* **Authentication workflows**: Correct credentials returning JWT tokens, incorrect logins throwing exceptions, and token revocation on logout.
* **Domain Invariants**: Verification of constructor exception throwing when domain constraints are violated.
* **Authorization checks & Repository behavior**: Mocking repository structures to verify queries return correct matching parameters.
* **API controller endpoint responses**: Inspecting returned action results and status codes.

Only code that passed the updated test suite (34 tests total, all passing with zero warnings or errors) was integrated into the final codebase.
